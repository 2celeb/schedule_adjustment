# 不明点・検討事項

## ユーザー管理

## Q1: グループは1つのみ想定か、1ユーザーが複数グループに所属できるか？

### 提案
複数グループへの所属を許可するのが望ましい。例えば、同じユーザーが「サッカーチーム」と「勉強会」など異なるグループに参加するケースは十分に考えられる。データベース設計上も memberships テーブルで多対多の関係を表現しているため、技術的な追加コストは小さい。

### 回答
ユーザーは複数のグループに所属できるようにしてください。
また管理者も複数のグループを管理できるようにしてください。


## Q2: Owner は1グループに1名のみか、複数の Owner（副管理者）を設定できるか？

### 提案
初期リリースでは Owner は1名のみとし、シンプルな権限モデルで開始するのが良い。将来的に副管理者（Co-Owner）ロールを追加する拡張は、memberships テーブルの role カラムに値を追加するだけで対応可能。

### 回答
Owner は原則一名で管理者的な機能を扱える。 Member は閲覧、予定の入力、予定のコミット(活動日の設定)ができるようにしてくだしさい。
管理者の権限は要求が決まったら考えることをドキュメントに TODO タスクとして記入しておいてください。


## Q3: ユーザー認証はメールアドレス+パスワードか、Google アカウントのみか、Discord アカウントでもログイン可能か？

### 提案
Google OAuth をメインの認証手段とするのが合理的。Google カレンダー連携が主要機能であるため、Google アカウントでのログインが自然な導線になる。メール+パスワード認証も補助的に用意しておくと、Google アカウントを持たないユーザーにも対応できる。Discord OAuth は優先度を下げて将来対応としたい。

### 回答
パスワードは絶対に管理したくないので Googleアカウント認証としてください。 

ただしGoogleアカウントを連携したくないユーザーもいるので不要の場合は認証不要。
管理者はBot の導入等に Discord の認証は必須、
管理者がGoogleアカウントを連携した場合はイベント用のカレンダーを作成できる仕様にしてください。

## Q13: 将来的に有料プラン（広告非表示）を導入する予定はあるか？

### 提案
最大20名の小規模利用を前提としているため、有料プランの需要は限定的かもしれない。ただし、広告非表示オプションを将来的に追加できるよう、Ad_Module の表示/非表示をユーザー設定で制御できる設計にしておくのが良い。

**「1人が払えばグループ全員の広告が消える」方式について:**

技術的には問題なく実現可能。グループに有料ユーザーが1人でもいれば、そのグループページで全員の広告を非表示にするだけ。

- Discord のサーバーブーストと同じ発想で、ユーザー心理として機能しやすい
- 20人規模のグループで1人分の課金しか発生しないが、そもそも小規模サービスの広告収入は月に数百円にもならない可能性が高い
- 月額300〜500円のグループプランであれば、広告収入の減少分は十分カバーできる
- 「個人課金」より「グループ課金」の方がこのサービスの性質に合っている

価格設定案:
- 無料: 広告あり
- グループプラン（月額300〜500円）: グループ内の誰か1人が支払えば全員の広告非表示

初期リリースでは広告のみで運用し、ユーザーからのフィードバックを見て有料プランを導入するのが低リスク。Ad_Module の表示/非表示をグループ設定で制御できる設計にしておけば、後から対応可能。

### 回答
まずは無料ユーザーを想定して作成しますが
将来的に有料ユーザーにも対応したいのでグループごとに広告の ON/ OFF を制御できる仕組みを実装してください。
有料会員については plan のどこかにTODOメモを残しておいてください。

## 認証・ユーザー識別

## Q16: パスワードなしで個人を識別しスケジュール入力させる方式はどうするか？

### 提案
以下の3つのアプローチが考えられる。

**方式A: 個人用トークンリンク（参考サイト character-sheets.appspot.com と同様）**
- 各メンバーに固有のURL（例: `/schedule/edit?token=abc123xyz`）を発行
- URLを知っている人＝本人として扱う
- 最もシンプルだがURLの漏洩リスクあり

**方式B: Discord OAuth 認証**
- Discord にログイン済みなら1クリックで認証完了（パスワード入力不要）
- Discord ユーザーIDで個人を識別。なりすまし不可
- Discord アカウントが必須

**方式C: ハイブリッド方式（推奨）**
- 基本はトークンリンクで手軽にアクセス
- Google カレンダー連携が必要なユーザーのみ Google OAuth で認証
- Discord Bot が各メンバーに個別リンクを DM で配布
- セキュリティと利便性のバランスが良い

Discord コミュニティでの利用を前提とするなら、方式C のハイブリッド方式が最適と考える。普段の予定入力はトークンリンクで手軽に、カレンダー連携など高度な機能を使う場合のみ OAuth 認証を求める形。

### 回答
Q18が実装できるのでこの質問は不要ではないでしょうか？

## Q18: ユーザー名選択によるゆるい識別方式の詳細設計

### 提案
参考サイト（キャラクターシート倉庫のスケジュール機能）と同様に、認証なし・パスワードなしで以下の方式を採用する。

**基本フロー:**
1. グループのスケジュールページは1つの共通URL（例: `/schedule/group/abc123`）
2. ページ上部にメンバー名が横並びで表示される（参考画像の「ちたん」「りふぃ」「いおりん」...のように）
3. ユーザーが自分の名前をクリック/タップすると、その人の入力モードに切り替わる
4. 選択したユーザーはブラウザの localStorage に保存し、次回アクセス時は自動で同じユーザーとして開く
5. 別のユーザーに切り替えたい場合はいつでもヘッダーから変更可能

**ゆるい識別のポリシー:**
- パスワードやOAuth認証は不要（Google カレンダー連携時のみ Google OAuth を使用）
- 他人が別のユーザーの予定を変更することも技術的には許容する（厳密なアクセス制御はしない）
- ただし変更時に以下の情報を履歴として記録し、抑止力とする:
  - User-Agent（OS / ブラウザ情報）
  - IP アドレスから推定した地域情報（例: 名古屋）
  - 変更日時
  - 変更内容（変更前 → 変更後）

**マルチデバイス対応:**
- localStorage はデバイスごとに独立するため、PC とスマートフォンそれぞれで初回にユーザーを選択する必要がある
- 一度選択すれば、各デバイスで次回以降は自動的に同じユーザーとして開く

**技術的なメリット:**
- 認証基盤が不要（Devise, JWT, OAuth 等のユーザー認証が不要）
- サーバー側の実装がシンプル
- ユーザーの導入障壁が極めて低い（URLを開いて名前をタップするだけ）

**Google カレンダー連携との関係:**
- Google カレンダー連携を使いたいユーザーのみ、Google OAuth で認証する
- OAuth で取得したトークンはサーバー側でユーザーに紐付けて暗号化保存
- Google カレンダー連携しないユーザーは完全に認証不要

### 回答
提案通りでOKです。

## Q20: Q8 の回答について、ページを開いたときに全メンバーの Google カレンダーを再取得する方式の場合、Google API のレート制限に注意が必要。20人分を一度に取得する運用は可能か？

### 提案
Google Calendar FreeBusy API は1リクエストで複数ユーザーの空き状況を一括取得できる（最大50カレンダー）。20人分であれば1リクエストで取得可能。

ただし以下の制約がある:
- FreeBusy API を使うには、各ユーザーが事前に Google OAuth で認証し、カレンダーへのアクセス権を付与している必要がある
- Google カレンダー未連携のメンバーは対象外（手動入力のみ）
- API の無料枠は 1,000,000 リクエスト/日なので、20人グループが数十個あっても問題ない

**提案するフロー:**
1. ユーザーがスケジュールページを開く
2. サーバーがキャッシュの有効期限を確認（例: 15分以内なら再取得しない）
3. キャッシュ切れの場合、Google カレンダー連携済みメンバー全員分を FreeBusy API で一括取得
4. 結果をキャッシュに保存し、Availability_Board に反映
5. 「今すぐ同期」ボタンでキャッシュを無視して強制再取得も可能

### 回答
提案通り Google OAuth 認証が行っている場合の予定を取得してください

## Q23: グループへのメンバー追加はどのように行うか？Owner が管理画面から追加するのか、招待リンクを共有するのか？

### 提案

## Q30: ◎○△× の記号表現は海外ユーザーにも通じるか？国際対応が必要か？

### 提案

**日本・韓国:**
- ○=正解/OK、×=不正解/NG は共通認識。△=条件付きも日本では広く理解される
- 韓国でも○×の意味は日本と同じ（東アジア共通の記号体系）

**中国・台湾:**
- ○×の記号体系は日本ほど浸透していない
- 中国では ✓（チェックマーク）=正解、✗=不正解が一般的（欧米寄りの慣習）
- 台湾は日本統治時代の影響で○×を理解する人もいるが、若い世代は ✓✗ が主流

**欧米（米・欧）:**
- ✓=YES/正解、✗=NO/不正解が標準。○は特別な意味を持たない
- △は記号としての意味がほぼない
- 注意: 日本では ✓ が「不正解」のマークとして使われることがあり、真逆の意味になる

**国際的なスケジュールツールの事例（Doodle 等）:**
- ✓（Yes）、?（If need be）、✗（No）の3段階
- 色による区別（緑=OK、黄=条件付き、赤=NG）が最も国際的に通じやすい

**提案する対応方針:**

初期リリースは日本語ユーザーを対象とするため ◎○△× で問題ない。将来的に国際対応する場合は以下の方式が有効:

- 内部データは数値（例: 2=◎, 1=○, 0=△, -1=×）で管理
- 表示は色（緑/黄緑/黄/赤）+ アイコンで区別
- ロケール設定で記号を切り替え可能にする:
  - 日本語: ◎ ○ △ ×
  - 英語: ✓✓ ✓ ? ✗
  - 共通: 色分け（緑/黄緑/黄/赤）

データモデルを記号に依存させなければ、後から国際対応を追加するコストは小さい。

**◎** と**✓✓** は不採用としてください。

## Q40: Discord / Google OAuth の情報とユーザーの紐付けは localStorage だけで十分か？

### 提案
localStorage だけでは不十分。以下の問題がある:

- 別デバイスからアクセスすると Google 連携が引き継がれない
- ブラウザのデータ消去で紐付けが失われる
- シークレットモードでは localStorage が使えない

Google OAuth トークンはサーバー側の `users` テーブルに保存されているが、「このブラウザのユーザー = サーバー上のどの users レコードか」を localStorage の `selectedUserId` だけに頼っているため、localStorage が消えると Google 連携済みアカウントにアクセスできなくなる。

**解決策: 認証レベルに応じた2層構造**

| レベル | 識別方法 | 対象 | デバイス間共有 |
|---|---|---|---|
| ゆるい識別 | localStorage のみ | Google 未連携メンバー | 不可（デバイスごとに選択） |
| OAuth 識別 | HttpOnly Secure Cookie | Google 連携メンバー / Owner | 可（再認証で紐付け） |

**フロー:**

1. **Google 未連携メンバー**: 従来通り localStorage でユーザーIDを記憶。Cookie 不要
2. **Google 連携メンバー**: Google OAuth 認証完了時にサーバーがセッション Cookie（HttpOnly, Secure, SameSite=Lax）を発行。以降は Cookie でユーザーを識別。別デバイスでも Google で再認証すれば同じユーザーに紐付く
3. **Owner（Discord OAuth 済み）**: Discord OAuth 認証完了時にサーバーがセッション Cookie を発行。管理機能へのアクセスは Cookie で認証

**Cookie の仕様:**
- HttpOnly: JavaScript からアクセス不可（XSS 対策）
- Secure: HTTPS 通信のみ
- SameSite=Lax: CSRF 対策
- 有効期限: 30日（更新可能）
- セッショントークンはサーバー側で生成し、Redis or DB に保存

**localStorage との併用:**
- Cookie がある場合: Cookie のセッションからユーザーを特定（優先）
- Cookie がない場合: localStorage の selectedUserId を使用（ゆるい識別にフォールバック）
- Google 連携済みユーザーが localStorage しかない状態でアクセスした場合: 「Google アカウントで再認証」ボタンを表示

### 回答
41の方法を採用してください。


## Q41: Q40 の2層構造で、別デバイスから異なる Google アカウントで認証した場合の競合問題をどう防ぐか？

### 提案
Q40 の設計では、ゆるい識別（誰でもユーザーを選べる）と OAuth 認証（本人確認）が矛盾する。別の人が「えれん」を選んで異なる Google アカウントで認証すると、えれんの連携が壊れる。

**解決策: Google 連携したユーザーは「ゆるい識別」から卒業させる**

Google 連携した時点で、そのユーザーは「Google アカウント = 本人確認の鍵」になる。

**UI の変化:**
- Google 未連携メンバー: 名前をクリックするだけで操作可能（従来通り）
- Google 連携済みメンバー: 名前の横に 🔒 アイコン表示。クリックすると「Google でログイン」ボタンが表示され、認証成功時のみ操作可能

**ルール:**
- 1つの users レコードに紐付く Google アカウントは1つだけ
- 連携済みユーザーに対して別の Google アカウントで認証しようとした場合 → エラー表示
- 別デバイスでも同じ Google アカウントで認証すれば同じユーザーとして操作可能
- Google 連携の解除は本人（設定画面から）または Owner（管理画面から強制解除）が可能
- 連携解除するとゆるい識別に戻る

**フロー例（えれんさん）:**
```
1. PC: 「えれん」を選択（ゆるい識別）→ 予定入力可能
2. PC: Google 連携ボタンを押す → Google アカウント A で認証
   → users[42] に Google A を紐付け、Cookie 発行
   → 以降「えれん」は 🔒 付き表示になる
3. モバイル: 「えれん 🔒」をタップ → 「Google でログイン」画面
4. モバイル: Google アカウント A で認証 → OK、Cookie 発行、えれんとして操作可能
5. モバイル: Google アカウント B で認証 → エラー「えれんは別のGoogleアカウントで連携済みです」
```

**Owner（Discord OAuth）も同様:**
- Discord OAuth で認証した Owner は 🔒 付き
- 管理機能へのアクセスは Cookie（Discord OAuth セッション）で認証
- 別デバイスでも同じ Discord アカウントで再認証すれば OK

**データモデルへの影響:**
- `users` テーブルに `google_account_id`（Google のユニークID）を追加
- `google_account_id` に UNIQUE 制約を付けて、1つの Google アカウントが複数ユーザーに紐付くのを防止
- `users` テーブルに `auth_locked`（boolean, default: false）を追加。Google 連携時に true にする
