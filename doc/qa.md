# 不明点・検討事項

## ユーザー管理

## Q1: グループは1つのみ想定か、1ユーザーが複数グループに所属できるか？

### 提案
複数グループへの所属を許可するのが望ましい。例えば、同じユーザーが「サッカーチーム」と「勉強会」など異なるグループに参加するケースは十分に考えられる。データベース設計上も memberships テーブルで多対多の関係を表現しているため、技術的な追加コストは小さい。

### 回答
ユーザーは複数のグループに所属できるようにしてください。
また管理者も複数のグループを管理できるようにしてください。


## Q2: Owner は1グループに1名のみか、複数の Owner（副管理者）を設定できるか？

### 提案
初期リリースでは Owner は1名のみとし、シンプルな権限モデルで開始するのが良い。将来的に副管理者（Co-Owner）ロールを追加する拡張は、memberships テーブルの role カラムに値を追加するだけで対応可能。

### 回答
Owner は原則一名で管理者的な機能を扱える。 Member は閲覧、予定の入力、予定のコミット(活動日の設定)ができるようにしてくだしさい。
管理者の権限は要求が決まったら考えることをドキュメントに TODO タスクとして記入しておいてください。


## Q3: ユーザー認証はメールアドレス+パスワードか、Google アカウントのみか、Discord アカウントでもログイン可能か？

### 提案
Google OAuth をメインの認証手段とするのが合理的。Google カレンダー連携が主要機能であるため、Google アカウントでのログインが自然な導線になる。メール+パスワード認証も補助的に用意しておくと、Google アカウントを持たないユーザーにも対応できる。Discord OAuth は優先度を下げて将来対応としたい。

### 回答
パスワードは絶対に管理したくないので Googleアカウント認証としてください。 

ただしGoogleアカウントを連携したくないユーザーもいるので不要の場合は認証不要。
管理者はBot の導入等に Discord の認証は必須、
管理者がGoogleアカウントを連携した場合はイベント用のカレンダーを作成できる仕様にしてください。

### 提案
過去の日付は閲覧のみ（ロック）にするのが自然。過去データの改変を防ぎ、活動履歴としての信頼性を保てる。

## Q7: Auto_Schedule_Rule は複数設定可能か？（例：毎週土曜日 + 毎月第1日曜日）

複数ルールの組み合わせを許可する方が柔軟で実用的。ただし UI の複雑さを抑えるため、最大3〜5ルール程度の上限を設けるのが良い。

## Q13: 将来的に有料プラン（広告非表示）を導入する予定はあるか？

### 提案
最大20名の小規模利用を前提としているため、有料プランの需要は限定的かもしれない。ただし、広告非表示オプションを将来的に追加できるよう、Ad_Module の表示/非表示をユーザー設定で制御できる設計にしておくのが良い。

**「1人が払えばグループ全員の広告が消える」方式について:**

技術的には問題なく実現可能。グループに有料ユーザーが1人でもいれば、そのグループページで全員の広告を非表示にするだけ。

- Discord のサーバーブーストと同じ発想で、ユーザー心理として機能しやすい
- 20人規模のグループで1人分の課金しか発生しないが、そもそも小規模サービスの広告収入は月に数百円にもならない可能性が高い
- 月額300〜500円のグループプランであれば、広告収入の減少分は十分カバーできる
- 「個人課金」より「グループ課金」の方がこのサービスの性質に合っている

価格設定案:
- 無料: 広告あり
- グループプラン（月額300〜500円）: グループ内の誰か1人が支払えば全員の広告非表示

初期リリースでは広告のみで運用し、ユーザーからのフィードバックを見て有料プランを導入するのが低リスク。Ad_Module の表示/非表示をグループ設定で制御できる設計にしておけば、後から対応可能。

### 回答
まずは無料ユーザーを想定して作成しますが
将来的に有料ユーザーにも対応したいのでグループごとに広告の ON/ OFF を制御できる仕組みを実装してください。
有料会員については plan のどこかにTODOメモを残しておいてください。

## 認証・ユーザー識別

## Q16: パスワードなしで個人を識別しスケジュール入力させる方式はどうするか？

### 提案
以下の3つのアプローチが考えられる。

**方式A: 個人用トークンリンク（参考サイト character-sheets.appspot.com と同様）**
- 各メンバーに固有のURL（例: `/schedule/edit?token=abc123xyz`）を発行
- URLを知っている人＝本人として扱う
- 最もシンプルだがURLの漏洩リスクあり

**方式B: Discord OAuth 認証**
- Discord にログイン済みなら1クリックで認証完了（パスワード入力不要）
- Discord ユーザーIDで個人を識別。なりすまし不可
- Discord アカウントが必須

**方式C: ハイブリッド方式（推奨）**
- 基本はトークンリンクで手軽にアクセス
- Google カレンダー連携が必要なユーザーのみ Google OAuth で認証
- Discord Bot が各メンバーに個別リンクを DM で配布
- セキュリティと利便性のバランスが良い

Discord コミュニティでの利用を前提とするなら、方式C のハイブリッド方式が最適と考える。普段の予定入力はトークンリンクで手軽に、カレンダー連携など高度な機能を使う場合のみ OAuth 認証を求める形。

Q18が実装できるのでこの質問は不要ではないでしょうか？

## Q18: ユーザー名選択によるゆるい識別方式の詳細設計

### 提案
参考サイト（キャラクターシート倉庫のスケジュール機能）と同様に、認証なし・パスワードなしで以下の方式を採用する。

**基本フロー:**
1. グループのスケジュールページは1つの共通URL（例: `/schedule/group/abc123`）
2. ページ上部にメンバー名が横並びで表示される（参考画像の「ちたん」「りふぃ」「いおりん」...のように）
3. ユーザーが自分の名前をクリック/タップすると、その人の入力モードに切り替わる
4. 選択したユーザーはブラウザの localStorage に保存し、次回アクセス時は自動で同じユーザーとして開く
5. 別のユーザーに切り替えたい場合はいつでもヘッダーから変更可能

**ゆるい識別のポリシー:**
- パスワードやOAuth認証は不要（Google カレンダー連携時のみ Google OAuth を使用）
- 他人が別のユーザーの予定を変更することも技術的には許容する（厳密なアクセス制御はしない）
- ただし変更時に以下の情報を履歴として記録し、抑止力とする:
  - User-Agent（OS / ブラウザ情報）
  - IP アドレスから推定した地域情報（例: 名古屋）
  - 変更日時
  - 変更内容（変更前 → 変更後）

**マルチデバイス対応:**
- localStorage はデバイスごとに独立するため、PC とスマートフォンそれぞれで初回にユーザーを選択する必要がある
- 一度選択すれば、各デバイスで次回以降は自動的に同じユーザーとして開く

**技術的なメリット:**
- 認証基盤が不要（Devise, JWT, OAuth 等のユーザー認証が不要）
- サーバー側の実装がシンプル
- ユーザーの導入障壁が極めて低い（URLを開いて名前をタップするだけ）

**Google カレンダー連携との関係:**
- Google カレンダー連携を使いたいユーザーのみ、Google OAuth で認証する
- OAuth で取得したトークンはサーバー側でユーザーに紐付けて暗号化保存
- Google カレンダー連携しないユーザーは完全に認証不要

### 回答
提案通りでOKです。
