# プロジェクト計画書

## 概要

小規模グループ（最大約20名）向けのスケジュール調整ツールを開発する。Discord コミュニティでの利用を前提とし、メンバーの参加可否を可視化、活動日の自動・手動設定、Discord Bot および Google カレンダーとの連携機能を提供する。パスワード不要の「ゆるい識別方式」を採用し、導入障壁を極力下げる。広告収入による運用を前提とし、低コスト構成で実現する。

## ユーザーロールと権限

| ロール | 説明 | 主な権限 |
|---|---|---|
| Owner | グループ管理者（1グループに1名） | 全機能。活動日の確定・変更、設定変更、メンバー管理、過去日付の編集 |
| Core_Member | 主要メンバー | 予定入力、閲覧、コメント。Threshold_N の判定対象 |
| Sub_Member | 補助メンバー | 予定入力、閲覧、コメント |

## プロジェクトフェーズ

### フェーズ 1: 基盤構築（2〜3週間）

- Ruby on Rails プロジェクトのセットアップ（API モード）
- React + Vite フロントエンドプロジェクトのセットアップ
- データベース設計とマイグレーション作成
- Docker Compose による開発環境構築
- CI/CD パイプラインの構築
- ゆるい識別方式の基盤実装（localStorage + ユーザー選択 UI）

### フェーズ 2: Discord Bot 基盤 + グループ管理（2〜3週間）

- Discord Bot の基盤開発（discord.js）
- Server Members Intent によるメンバーリスト自動取得
- `/schedule` コマンド（初回設定 + スケジュールURL表示）
- `/status` コマンド（今週の入力状況表示）
- `/settings` コマンド（設定画面URL表示、Owner のみ）
- グループ基本設定の保存（グループ名、イベント名、基本活動時間、タイムゾーン）
- Owner の Discord OAuth 認証
- メンバー表示名の変更機能（Discord スクリーン名はホバーで表示）

### フェーズ 3: コア機能開発（3〜4週間）

- Availability_Board の UI 実装（PC + スマートフォン対応）
  - 月単位表示（デフォルト）+ 週単位切り替え
  - メンバー名横並び表示、クリックで入力モード切り替え
  - localStorage によるユーザー記憶
- 参加可否の入力・保存（○/△/×/−）
  - 内部データは数値管理（国際化対応）
  - ロケール設定による記号切り替え（日本語: ○△×、英語: ✓?✗）
  - 色分け表示（緑/黄/赤/グレー）
- コメント入力機能（×/△の日にコメント付与）
- 集計表示（各日の○/△/×/−の人数）
- Core_Member / Sub_Member の区別表示
- Threshold_N の設定と警告色表示
- 過去日付のロック（Owner のみ編集可能）
- 変更履歴の記録（User-Agent、IP ベース地域情報、変更日時、変更内容）

### フェーズ 4: 活動日設定・自動確定機能（3週間）

- 手動での活動日設定（Owner のみ）
- Auto_Schedule_Rule の設定 UI と自動生成ロジック
  - 最大活動日数/週、最低活動日数/週
  - 優先度を下げる曜日（デフォルト: なし）
  - 除外曜日（最低活動日数未達の場合は活動日とする）
  - 活動日判定基準（△×がない人は活動日とする）
- 週の始まり設定（全曜日選択可、デフォルト: 月曜日）
- 自動確定タイミング設定（デフォルト: 週の始まりの3日前）
- リマインド機能
  - リマインド開始日（デフォルト: 確定日の2日前）
  - 1回目: チャンネルにメンション
  - 2回目（翌日）: DM で個別通知
- 確定後の Discord チャンネルへの予定一覧投稿
- 日々の活動時間の個別変更（Owner のみ、変更箇所は赤の！マークで強調）
- 活動日当日の通知（デフォルト: 活動開始8時間前、メンションなし、チャンネル設定可能）

### フェーズ 5: Google カレンダー連携（3〜4週間）

- Owner の Google OAuth 認証（書き込み権限: `calendar` スコープ）
- Owner の Google カレンダーにサブカレンダー自動作成（「[グループ名] イベント」）
- メンバーの Google カレンダー連携（3パターン、メンバー自身が選択）
  - 連携なし
  - 予定枠のみ連携（FreeBusy: 読み取り専用）
  - 予定枠 + 書き込み連携（FreeBusy + イベント作成）
- ページ表示時の FreeBusy API 一括取得（キャッシュ15分、20人分1リクエスト）
- 「今すぐ同期」ボタン
- 予定がある日の自動×設定（終日・時間指定とも一律×、手動で変更可能）
- 活動日確定時の Google カレンダー書き込み
  - Owner のサブカレンダーに予定追加（参加/不参加メンバー一覧付き）
  - 書き込み連携メンバーの個人カレンダーに予定作成

### フェーズ 6: 広告・運用準備（1〜2週間）

- Google AdSense の導入
- 広告配置の最適化
  - デスクトップ: ヘッダーまたはサイドバーにバナー
  - モバイル: フッター固定バナー
  - 入力中はポップアップ/インタースティシャル広告を表示しない
- グループ単位の広告 ON/OFF 制御機能
- パフォーマンス最適化
- セキュリティレビュー

### フェーズ 7: テスト・リリース（2週間）

- 統合テスト・E2E テスト
- ユーザー受け入れテスト
- 本番環境デプロイ
- 運用監視の設定

## リスクと対策

| リスク | 影響度 | 対策 |
|---|---|---|
| Google API の利用制限 | 中 | FreeBusy API で一括取得。15分キャッシュ。無料枠は100万リクエスト/日で十分 |
| Discord Server Members Intent の審査 | 低 | 100サーバー未満は審査不要。小規模利用を前提 |
| ゆるい識別方式のなりすまし | 低 | 変更履歴（OS・地域情報）を記録し抑止力とする。リスクは許容範囲 |
| 広告収入の不足 | 中 | 初期は ConoHa VPS 2GB プラン（月1,259円）で運用。広告 + 有料プランで回収を目指す |
| プライバシー問題 | 高 | 予定の詳細を一切保存しない設計。FreeBusy のブール値のみキャッシュ |
| VPS の単一障害点 | 中 | Docker の restart policy で自動復旧。Uptime Kuma + Discord Webhook で障害検知 |
| 開発遅延 | 中 | フェーズごとに MVP を定義し段階的にリリース |

## 成功指標

- Discord Bot 導入からスケジュール入力開始まで5分以内で完了できること
- Availability_Board の表示が1秒以内であること
- Google カレンダー連携の設定が3ステップ以内で完了できること
- 月間サーバーコストが 2,500円 以下であること（初期）
- リマインド送信後24時間以内の入力率が80%以上であること
